# v0.195.495 — Outliner: collapsedノードでの↑↓ナビゲーション修正

## 課題

折りたたんだノード（`collapsed: true`）にカーソルがある状態で↓を押しても、次の表示行に移動しない。↑も同様。通常時・スコープ時・検索結果のいずれでも発生。

## 原因

v0.195.494で導入した`getDomNextNodeEl` / `getDomPrevNodeEl`は、`querySelectorAll('.outliner-node')`で全ノードを取得し、現在位置の次/前の要素を返す。しかし、collapsedノードの子は`.is-collapsed`クラスの付いたコンテナ内にあり、CSSで`display: none`されているが、`querySelectorAll`はDOMに存在する全要素を返すため**非表示の子ノードも含まれる**。

例:
```
index 0: NodeA (collapsed)     ← ここにいる
index 1: NodeA-child1          ← .is-collapsed内（非表示）
index 2: NodeA-child2          ← .is-collapsed内（非表示）
index 3: NodeB                 ← 次の表示行
```

`allNodes[i+1]`はindex 1のNodeA-child1（非表示）を返すため、フォーカスが飛ばず動かないように見える。

## 対策

### 設計

次/前の要素を探す際、候補が`.is-collapsed`の子孫かどうかを`closest('.is-collapsed')`でチェックし、非表示ノードをスキップする。

### 実装

`src/webview/outliner.js` の `getDomPrevNodeEl` / `getDomNextNodeEl`:

```javascript
// 変更前: 隣接要素を即座に返す
if (allNodes[i] === currentNodeEl) {
    return (i < allNodes.length - 1) ? allNodes[i + 1] : null;
}

// 変更後: 隣接要素が非表示ならスキップして次を探す
if (allNodes[i] === currentNodeEl) {
    for (var j = i + 1; j < allNodes.length; j++) {
        if (!allNodes[j].closest('.is-collapsed')) { return allNodes[j]; }
    }
    return null;
}
```

`closest('.is-collapsed')`は祖先を辿るため、ネストされたcollapse（折りたたみの中の折りたたみ）も正しくスキップされる。collapsedノード自体は`.is-collapsed`コンテナの**外**にあるためスキップされない。
