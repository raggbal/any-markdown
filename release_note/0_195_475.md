# v0.195.475 — SidePanelManager 共通化リファクタリング

## 課題

`editorProvider.ts`（mdモード）と `outlinerProvider.ts`（outlinerモード）の両方に、サイドパネルのファイル監視・保存・リンク処理・TOC抽出の完全に同一のロジックが約270行ずつ重複していた。修正時に片方を直して片方を忘れるリスクが高く、v0.195.472-474で画像貼り付け問題が発生した際にも、メッセージ形式の不一致（`{ html }` vs `{ markdownPath, displayUri }`）が重複コードに起因していた。

## 原因

outlinerモードにサイドパネル機能を追加した際（v0.195.470〜）、editorProviderからサイドパネル関連コードをコピー＆ペーストで移植した。同一ロジックが2ファイルに分散し、一方の修正が他方に反映されない構造だった。

## 対策

### 設計

`src/shared/sidePanelManager.ts` に `SidePanelManager` クラスを新規作成し、両プロバイダに共通するサイドパネルロジックを抽出。画像ハンドラは意図的に共通化しない（ディレクトリ解決ロジックが根本的に異なるため）。

**SidePanelManager が担当する機能:**
- ファイル監視 (TextDocument + FileSystemWatcher + onDidChangeTextDocument)
- 保存ハンドラ (`handleSave`)
- クローズハンドラ (`handleClose`)
- リンク処理 (`handleOpenLink`)
- ファイルオープン (`openFile` — ファイル読み込み→TOC抽出→メッセージ送信→ファイル監視設定)
- TOC抽出 (`extractToc` — static method)

**SidePanelManager が担当しない機能（各プロバイダに残す）:**
- 画像挿入 (`insertImage`, `saveImageAndInsert`, `readAndInsertImage`) — editor: ImageDirectoryManager + directives + settings / outliner: 固定 `{pageDir}/images/`
- 画像ディレクトリ状態送信 (`sendSidePanelImageDirStatus`) — 上記と同じ理由
- `setImageDir` — editor: 複雑なダイアログフロー / outliner: no-op

### 実装

1. **`src/shared/sidePanelManager.ts` 新規作成** (~275行)
   - `SidePanelHost` インターフェース: `postMessage` + `asWebviewUri`
   - `SidePanelManager` クラス: 6つの内部状態変数 + 7メソッド + 3アクセサ
   - アクセサ `watchedPath`, `document`, `isApplyingEdit` でeditorProviderの画像解決ヘルパーが内部状態を参照可能

2. **`src/outlinerProvider.ts` 移行** (-120行/+20行)
   - 状態変数6個、`setupSidePanelFileWatcher`, `disposeSidePanelFileWatcher`, `openFileInSidePanel`, `extractToc` を削除
   - メッセージハンドラを `sidePanel.openFile()`, `sidePanel.handleSave()`, `sidePanel.handleClose()`, `sidePanel.handleOpenLink()` に置換

3. **`src/editorProvider.ts` 移行** (-150行/+30行)
   - 同上の削除・置換
   - 画像ハンドラの `sidePanelWatchedPath` → `sidePanel.watchedPath`、`sidePanelDocument` → `sidePanel.document` に置換

### テスト結果

- `npm run compile`: コンパイル成功
- `npx playwright test`: 653 passed（失敗6件は既存テストで今回の変更と無関係）
