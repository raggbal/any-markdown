# v0.195.553 リリースノート

## Outliner: ノード追加位置が保存時に壊れるバグを修正

### 課題

Outliner editorでEnterやOption+Enterで兄弟・子ノードを追加した際、追加した位置で保存されない致命的バグ。
Notes editorの左サイドパネルで別の.outファイルに切り替えて再度開くと、追加したノードが親ノードの子の末尾に移動していた。
fractal化以降ずっと発生していた。

### 原因

`OutlinerModel._ensureChildren()` の `needsRebuild` 判定ロジックの欠陥。

```javascript
// 旧コード: リーフノード（children.length === 0）が存在すれば常にtrue
if (node.parentId && node.children.length === 0) {
    needsRebuild = true;
}
```

この条件は「parentIdを持ち、childrenが空のノード」で判定しているが、子を持たない末端ノード（リーフノード）は必ずこの条件に該当する。そのため、事実上全てのアウトライナーツリーで `needsRebuild = true` となり、**毎回全ノードのchildren配列がクリアされてJSオブジェクトのキー順で再構築されていた。**

新規追加ノードはJSオブジェクトの末尾にキーが追加されるため、再構築時に親のchildren末尾に配置されてしまう。

### 対策

#### 設計

- `_ensureChildren()` のchildren再構築ロジックを完全削除
- 旧形式データ（childrenプロパティが存在しないノード）は `console.error` でエラー表示のみ
- 現行フォーマットでは `serialize()` が全ノードに `children` 配列を含めて保存するため、再構築は不要

#### 実装

`src/webview/outliner-model.js` の `_ensureChildren()` を以下に変更:

- childrenプロパティが存在しないノードを検出した場合はエラーログ出力 + 空配列で初期化
- subtextデフォルト設定は維持
- children再構築ループ（needsRebuild判定 + クリア + parentIdベース再構築）を全て削除
