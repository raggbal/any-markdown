# v0.195.415 - サイドパネル画像問題の根本修正

## 問題の概要

サイドパネル（iframe）内エディタで画像が一切表示されない問題。
ペースト（Cmd+V）、ドラッグ＆ドロップ、ツールバー挿入のいずれも影響。

## 根本原因

サイドパネルは `blob:` URL で読み込まれたiframeとして動作する。
この blob: iframe は以下の URI スキームを一切解決できない:

| URI スキーム | 理由 |
|---|---|
| `https://file+.vscode-resource.vscode-cdn.net/...` | VSCode webview の内部プロトコルハンドラが blob: オリジンでは動作しない（`ERR_NAME_NOT_RESOLVED`） |
| `file:///...` | blob: iframe のセキュリティサンドボックスにより file: プロトコルへのアクセスが拒否される |
| `fetch()` で vscode-resource URI を取得 | CSP の `connect-src` に vscode-resource ドメインが含まれないため `Refused to connect` |

つまり、blob: iframe 内では **外部ファイルシステムの画像を直接参照する方法が存在しない**。

## 影響範囲

| 操作 | 症状 |
|---|---|
| サイドパネルを開く（既存画像あり） | 画像がすべて壊れたアイコンで表示 |
| D&D で画像挿入 | markdownリンクは挿入されるが画像は表示されない |
| Cmd+V で画像ペースト | markdownリンクは挿入されるが画像は表示されない |
| ソースモード切替後の再描画 | 画像が消える |
| サイドパネル内リンクでナビゲーション | 遷移先の画像も表示されない |

## 試行した解決策と失敗

### 試行1: `file://` URI への変換（v0.195.412）

`displayUri` を `vscode-resource` から `file://` に変換してiframeに送信。

**失敗理由**: blob: iframe は `file://` プロトコルにもアクセスできない。

### 試行2: main webview で `fetch()` → data URL 変換（v0.195.413）

main webview（vscode-resource にアクセス可能）で画像を fetch し、FileReader で data URL に変換してからiframeに送信。

**失敗理由**: CSP の `connect-src` に vscode-resource ドメインが含まれないため、fetch 自体がブロックされる（`Refused to connect because it violates the document's Content Security Policy`）。

### 解決策: editorProvider.ts でファイル読み取り → data URL 生成（v0.195.414-415）

**Node.js 側（editorProvider.ts）でファイルを直接読み取り**、base64 data URL を生成。
blob: iframe のブラウザ側制約を完全にバイパス。

## 最終的な修正内容

### 1. 画像マップ方式（初期レンダリング + 再レンダリング対応）

**editorProvider.ts**: `buildImageMap()` メソッド追加
- markdown 内の `![alt](path)` パターンを正規表現で抽出
- 各画像ファイルを `fs.readFileSync()` で読み取り
- `data:image/png;base64,...` 形式の data URL に変換
- `{パス: dataURL}` のマップを生成

**webviewContent.ts**: `getSidePanelHtml()` に `imageMap` パラメータ追加
- editor.js 内の `var sidePanelImageMap = {};` を `var sidePanelImageMap = {実際のマップ};` に置換
- iframe の初期化時点で imageMap が利用可能

**editor.js**: `resolveImagePath()` に imageMap 参照を追加
- 画像パスの解決時に `sidePanelImageMap[src]` を最初にチェック
- data URL が見つかればそれを使用（file:// や vscode-resource への変換をスキップ）
- 初期レンダリング・ソースモード切替・外部変更による再レンダリングすべてで動作

### 2. 新規画像挿入対応（D&D / Cmd+V / ツールバー）

**editorProvider.ts**: `handleSaveImage()` / `handleReadAndInsertImage()` / `handleInsertImage()`
- `insertImageHtml` メッセージに `dataUri` フィールドを追加
- 画像保存後、同じファイルを読み取って data URL を生成

**editor.js**: iframe リレー時に `dataUri` を使用
- `sidePanelImagePending` フラグで判定
- `message.dataUri || message.displayUri` で data URL 優先
- iframe 内で受信時に `sidePanelImageMap[markdownPath] = dataUri` でキャッシュ追加
- 以降の再レンダリングでも画像が維持される

### 3. `setImageMap` メッセージハンドラ（補助）

iframe load 後に main editor.js から imageMap を送信し、既存画像の src を更新する仕組みも追加（HTML 埋め込みのバックアップとして）。

## 修正ファイル

| ファイル | 変更内容 |
|---|---|
| `src/editorProvider.ts` | `buildImageMap()` 追加、3箇所の `insertImageHtml` に `dataUri` 追加、`getSidePanelHtml()` に imageMap 渡し |
| `src/webviewContent.ts` | `getSidePanelHtml()` に `imageMap` パラメータ追加、editor.js への埋め込み |
| `src/webview/editor.js` | `sidePanelImageMap` 変数追加、`resolveImagePath()` に imageMap 参照追加、`setImageMap` ハンドラ追加、iframe リレーで dataUri 使用 |

## 教訓

blob: iframe は外部リソースへのアクセスが極めて制限されている。`file://`、`vscode-resource`、`fetch()` のいずれも使用できない。iframe に外部画像を表示するには、**ホスト側（Node.js）でファイルを読み取り、data URL として埋め込む**のが唯一確実な方法。
