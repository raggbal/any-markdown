# v0.195.494 — Outliner: 検索結果での↑↓矢印キーナビゲーション修正

## 課題

検索結果（ツリーモード）で↑↓矢印キーを押すと、次/前の行ではなく、DOM上で離れた別の行にフォーカスが飛ぶことがある。特に異なる親ツリーの境界を跨ぐ際に頻発する。

## 原因

検索ツリーモードでは、同一ノードがマッチ結果と祖先コンテキストの両方で描画される。これにより**DOMに同じ`data-id`を持つ要素が複数存在**する（例: 142ノード中、多数のIDが重複）。

以前の実装には2つの問題があった:

1. **`getDomNextNodeId` / `getDomPrevNodeId`**: `querySelectorAll`で全ノードを走査し、`data-id`でマッチ→次のIDを返す方式。しかし重複IDがある場合、最初にヒットした要素を基準にしてしまい、カーソルが実際にいるDOM要素とは異なる位置の要素を見つける可能性があった。

2. **`focusNode`**: IDで`querySelector`を実行するため、重複IDの**最初のマッチ**（DOM上部の祖先コンテキスト側）にフォーカスが当たる。ユーザーが下方の検索結果ノードにいるのに、上方の同一IDノードにジャンプしてしまう。

## 対策

### 設計

IDベースのルックアップを廃止し、**DOM要素の同一性比較（`===`）**に切り替える。

### 実装

`src/webview/outliner.js`:

1. **`getDomPrevNodeEl(currentTextEl)` / `getDomNextNodeEl(currentTextEl)`**: IDではなく、現在の`.outliner-text`要素から`.closest('.outliner-node')`で親ノード要素を取得。`querySelectorAll`で全ノードを取得し、`allNodes[i] === currentNodeEl`（オブジェクト同一性）で現在位置を特定。前後のDOM要素を直接返す。

2. **`focusNodeEl(nodeEl)` / `focusNodeElAtStart(nodeEl)`**: DOM要素を直接受け取り、`querySelector`によるIDルックアップを回避。`nodeEl.querySelector('.outliner-text')`で直接テキスト要素を取得してフォーカス。

3. **矢印キーハンドラ**: `getDomNextNodeEl(textEl)` → `focusNodeEl(nextEl)` のチェーンに変更。IDを介さずDOM要素を直接渡すため、重複IDの影響を受けない。

4. **`focusFirstVisibleNode()`**: `focusNodeElAtStart(firstNodeEl)` に変更し、DOM要素直接方式に統一。

### 修正のポイント

- `querySelector`はCLAUDE.md Section 16の通り「最初のマッチしか返さない」。重複IDが存在するDOMではIDベースのルックアップが根本的に不適切。
- DOM要素の同一性比較（`===`）は重複IDに影響されず、常に正しい位置を特定できる。
