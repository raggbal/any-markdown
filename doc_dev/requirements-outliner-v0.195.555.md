## 要件1: Shift+↑/↓ 初回選択の修正

### 現状の問題

Shift+↑ または Shift+↓ で複数行選択を開始する際、初回キー押下で **2行** が選択されてしまう。

**原因**: 初回Shift+↑押下時、`selectionAnchorId` を現在行に設定し、直後に `selectRange(現在行, 前の行)` を呼ぶ。`selectRange` は inclusive range（fromとtoの両方を含む）なので、現在行と前の行の2行が選択される。

### 要件

| No | 要件 |
|---|---|
| SEL-1 | Shift+↑/↓ の初回押下では、**自行のみ** を行選択状態にする（2行選択しない） |
| SEL-2 | 初回押下でフォーカスは移動しない（自行にとどまる） |
| SEL-3 | 2回目以降のShift+↑/↓ は従来通り、選択範囲を1行ずつ拡張する |
| SEL-4 | Shift+↑で上方向に拡張、Shift+↓で下方向に拡張する動作は維持 |
| SEL-5 | 既に行選択状態にある場合（`selectionAnchorId` が設定済み）は従来動作のまま |

### 期待する操作フロー（Shift+↑の場合）

```
初期状態: フォーカスが行C（未選択）
  Shift+↑ 1回目 → 行Cのみ選択（アンカー=C）、フォーカスはCのまま
  Shift+↑ 2回目 → 行B〜C選択、フォーカスはBに移動
  Shift+↑ 3回目 → 行A〜C選択、フォーカスはAに移動
```

## 要件2: ページノードのクリップボード操作（page保持）

### 現状の問題

複数行選択してCmd+C / Cmd+Xし、Cmd+Vで貼り付けた際、ページ属性（`isPage`, `pageId`）を持つノードの **ページ情報が消失** する。

**原因**:
- `getSelectedText()` はプレーンテキスト（`node.text`）のみをクリップボードにコピー
- `pasteNodesFromText()` は `model.addNode()` で新規ノードを作成するが、`isPage: false, pageId: null` で初期化される
- ページメタデータを保持する内部クリップボード機構が未実装

### 要件

| No | 要件 |
|---|---|
| PG-1 | Cmd+C / Cmd+X で複数行選択をコピー/カット時、ページメタデータ（`isPage`, `pageId`）を内部クリップボードに保存する |
| PG-2 | Cmd+V 貼り付け時、内部クリップボードにページメタデータがあればそれを復元する |
| PG-3 | **Cmd+X → Cmd+V（カット→ペースト）**: ページノードの `pageId` はそのまま引き継ぐ。元の `.md` ファイルをそのまま使う（移動扱い） |
| PG-4 | **Cmd+C → Cmd+V（コピー→ペースト）**: ページノードごとに新しい `pageId` を発行し、元の `.md` ファイルを新 `pageId` のファイル名で複製する |
| PG-5 | PG-4の`.md`ファイル複製は、ペースト対象のページノード数分すべて実行する |
| PG-6 | システムクリップボードにはプレーンテキストを書き込む（外部アプリとの互換性） |
| PG-7 | 内部クリップボードとシステムクリップボードのテキストが一致しない場合（外部からのペースト）、内部クリップボードは無視してプレーンテキストでペーストする |
| PG-8 | カット時の内部クリップボードは1回消費（ペースト後にクリア） |
| PG-9 | コピー時の内部クリップボードは何度でもペースト可能（ペースト毎に新pageId発行+ファイル複製） |

### 内部クリップボードデータ構造

```javascript
var internalClipboard = {
    plainText: "コピーされたテキスト",  // システムクリップボードとの照合用
    isCut: true/false,                  // カットかコピーか
    nodes: [
        { text: "ノードテキスト", level: 0, isPage: true, pageId: "uuid-xxx" },
        { text: "子ノード", level: 1, isPage: false, pageId: null },
        ...
    ]
};
```

### ホスト側の新規メッセージ

| メッセージ | 方向 | 内容 |
|---|---|---|
| `copyPageFile` | webview → host | `{ sourcePageId, newPageId }` — 元の `.md` を新 pageId でコピー |

## 要件3: 選択色のオレンジ化

### 現状の問題

行選択時の背景色と、行内テキスト範囲選択時の色が水色系で、現在行（フォーカス行）の背景色も水色のため **区別がつきにくい**。

### 要件

| No | 要件 |
|---|---|
| CLR-1 | **行選択（`.is-selected`）の背景色** をオレンジ系に変更する |
| CLR-2 | **テキスト範囲選択（`::selection`）の色** をオレンジ系に変更する（outliner内のみ） |
| CLR-3 | 現在行（`.is-focused`）の背景色は現状維持（水色系）とする。行選択のオレンジと明確に区別できるようにする |
| CLR-4 | 全7テーマ（github, sepia, night, dark, minimal, things, perplexity）で適用する |
| CLR-5 | ライトテーマではオレンジ系の薄い背景色、ダークテーマではオレンジ系の暗い背景色とする |

### 配色イメージ

| テーマ系統 | 行選択（.is-selected） | テキスト選択（::selection） |
|---|---|---|
| ライト系 | `rgba(255, 165, 0, 0.2)` 程度 | `rgba(255, 165, 0, 0.3)` 程度 |
| ダーク系 | `rgba(255, 165, 0, 0.15)` 程度 | `rgba(255, 165, 0, 0.25)` 程度 |

## 要件4: メニューボタンのフロートメニュー位置修正

### 現状の問題

検索バーの右端にあるメニューボタン（⋮）をクリックした際、フロートメニュー（ドロップダウン）が **メニューボタンの直下ではなく、画面右端** に表示されてしまう。

**原因**: ドロップダウンが `.outliner-search-bar`（フル幅）に `appendChild` され、CSSが `position: absolute; right: 0; top: 100%` で検索バー右端に配置。メニューボタンは検索バー右端にあるため概ね正しい位置だが、検索バーの `padding-right`（`var(--outliner-gutter)` = 32px）分だけ右にずれている。

### 要件

| No | 要件 |
|---|---|
| MENU-1 | フロートメニューをメニューボタンの直下に正確に配置する |
| MENU-2 | メニューの右端がメニューボタンの右端と揃うようにする |
| MENU-3 | 画面右端からはみ出さないようにする |
| MENU-4 | 既存のメニュー項目（Open in Text Editor, Copy File Path, Set page directory）の動作に影響しない |
